# AI 生成代码对比：有 Memory vs 无 Memory

## 维度对比

| 维度 | 有 Memory | 无 Memory |
|------|----------|-----------|
| **命名风格** | 遵循项目已有的命名规范（snake_case/PascalCase），与现有代码保持一致 | 可能使用默认或随机风格，与项目规范不一致 |
| **docstring** | 生成符合项目要求的 Google 风格文档字符串，包含 Args/Returns/Raises | 可能生成简略或不符合规范的注释 |
| **日志方式** | 使用项目指定的 logging 模块，拒绝裸 print() | 可能使用 print() 或不符合规范的日志方式 |
| **错误处理** | 遵循项目异常处理规范，使用项目定义的异常类型 | 可能使用过于宽泛的异常捕获或缺少错误处理 |
| **文件位置** | 根据项目结构将代码放在正确的目录位置 | 可能随意放置或不符合项目组织结构 |

## 结论

有 Memory 的 AI 辅助编码能够更好地理解并遵循项目的技术规范和编码约定。在有明确上下文指引的情况下，AI 生成的代码在命名规范、文档风格、日志使用、错误处理和文件组织等关键维度上都能与项目保持一致，显著减少后续的代码审查和重构工作。建议在重要项目中始终提供 Memory（项目规范、现有代码示例等）以获得更高质量的生成结果。
